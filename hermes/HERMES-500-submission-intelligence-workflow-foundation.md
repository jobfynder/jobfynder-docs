## 1. Module Purpose

HERMES-500 builds the first production foundation for submission intelligence inside Hermes.

This module connects the intelligence built in earlier Hermes modules to the real staffing workflow:

- jobs
- consultants
- recruiters
- bench sales recruiters
- introductions
- submissions
- submission stages
- follow-ups
- outcomes
- workflow history

The goal is not to build the full Jobfynder ATS or UI here.

The goal is to make Hermes understand and structure the submission workflow so Jobfynder can track, reason, and act on recruiting activity with clean intelligence.

---

## 2. Simple Definition

HERMES-500 answers this question:

> When a recruiter, bench sales recruiter, or consultant talks about a submission, what exactly is happening, what stage is it in, what action is needed next, and what should Jobfynder remember?

---

## 3. Scope

HERMES-500 includes:

1. Submission intelligence schema foundation
2. Submission lifecycle stage model
3. Workflow event model
4. Introduction and submission state transitions
5. Follow-up intelligence foundation
6. Outcome tracking foundation
7. Duplicate and conflict warning foundation
8. API contracts for submission intelligence
9. Parser-to-submission workflow handoff
10. Matching-to-submission workflow handoff
11. Taxonomy-to-submission explanation handoff
12. Basic tests and verification endpoints

---

## 4. Out of Scope

HERMES-500 does not include:

- Full UI implementation
- Full ATS replacement
- Payment logic
- Commission tracking
- Marketplace billing
- Advanced analytics dashboards
- Production-grade notification engine
- Full automation rules engine
- Multi-channel message ingestion expansion beyond existing Hermes foundations

These can come later after the workflow foundation is stable.

---

## 7. Required Design Principle

Keep HERMES-500 simple.

A 12th grader should be able to understand the workflow.

Hermes should not become a heavy ATS.

Hermes should act as the intelligence layer that knows:

- what happened
- who is involved
- what stage it is in
- what changed
- what needs attention
- what should be remembered

- workflow version

- allowed transitions
- terminal stages

- conflicts
- outcome
- reasons
- risks
- next actions
- handoff payload for Jobfynder API

## 14. Current HERMES-500 Foundation Behavior

The foundation currently supports:

1. Moving a discovered candidate/job pair to `matched` when matching output says the candidate should be submitted or reviewed.
2. Creating a workflow event when a stage change is recommended.
3. Creating follow-up instructions when a candidate is matchable.
4. Detecting duplicate-risk when a consultant/job key already exists.
5. Returning conflict details for duplicate-risk cases.
6. Returning a deterministic handoff payload for Jobfynder API.
7. Exposing workflow policy and evaluation contracts through the live API.

## 18. Current HERMES-500 API Surface

HERMES-500 currently exposes three submission intelligence endpoints.

### GET `/submissions/workflow-policy`

Returns the deterministic workflow policy.

Use this when Jobfynder needs to know:

- allowed stages
- allowed transitions
- terminal states
- active workflow version

### POST `/submissions/evaluate`

Evaluates a direct HERMES-500 submission intelligence request.

Use this when Jobfynder already has structured submission objects.

### POST `/submissions/evaluate/from-handoff`

Evaluates a handoff payload from earlier Hermes modules.

Use this when Jobfynder or Hermes has:

- parsed resume result
- parsed job result
- match result
- taxonomy context
- optional existing submission keys

This endpoint is the main bridge between understanding, matching, taxonomy, and workflow intelligence.
