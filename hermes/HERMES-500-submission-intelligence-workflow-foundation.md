## 1. Module Purpose

HERMES-500 builds the first production foundation for submission intelligence inside Hermes.

This module connects the intelligence built in earlier Hermes modules to the real staffing workflow:

- jobs
- consultants
- recruiters
- bench sales recruiters
- introductions
- submissions
- submission stages
- follow-ups
- outcomes
- workflow history

The goal is not to build the full Jobfynder ATS or UI here.

The goal is to make Hermes understand and structure the submission workflow so Jobfynder can track, reason, and act on recruiting activity with clean intelligence.

---

## 2. Simple Definition

HERMES-500 answers this question:

> When a recruiter, bench sales recruiter, or consultant talks about a submission, what exactly is happening, what stage is it in, what action is needed next, and what should Jobfynder remember?

---

## 3. Scope

HERMES-500 includes:

1. Submission intelligence schema foundation
2. Submission lifecycle stage model
3. Workflow event model
4. Introduction and submission state transitions
5. Follow-up intelligence foundation
6. Outcome tracking foundation
7. Duplicate and conflict warning foundation
8. API contracts for submission intelligence
9. Parser-to-submission workflow handoff
10. Matching-to-submission workflow handoff
11. Taxonomy-to-submission explanation handoff
12. Basic tests and verification endpoints

---

## 4. Out of Scope

HERMES-500 does not include:

- Full UI implementation
- Full ATS replacement
- Payment logic
- Commission tracking
- Marketplace billing
- Advanced analytics dashboards
- Production-grade notification engine
- Full automation rules engine
- Multi-channel message ingestion expansion beyond existing Hermes foundations

These can come later after the workflow foundation is stable.

## 6. Submission Lifecycle Draft

Initial lifecycle stages:

1. `discovered`
2. `matched`
3. `intro_requested`
4. `intro_accepted`
5. `submitted`
6. `screening`
7. `client_submitted`
8. `interview`
9. `offer`
10. `placed`
11. `rejected`
12. `withdrawn`
13. `duplicate_risk`
14. `closed_lost`

This stage model may be refined during implementation.

---

## 7. Required Design Principle

Keep HERMES-500 simple.

A 12th grader should be able to understand the workflow.

Hermes should not become a heavy ATS.

Hermes should act as the intelligence layer that knows:

- what happened
- who is involved
- what stage it is in
- what changed
- what needs attention
- what should be remembered

## 9. Open Decisions

These must be finalized during implementation:

1. Should Hermes persist submission workflow state directly, or only produce structured intelligence for Jobfynder API to persist?
2. Should duplicate detection be handled inside Hermes, Jobfynder API, or both?
3. Should follow-up rules be deterministic first, AI-assisted later, or mixed from the start?
4. Should submission IDs be generated by Hermes or only by Jobfynder API?
5. Should introductions and submissions be separate top-level objects or one unified workflow object?
